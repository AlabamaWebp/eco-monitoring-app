# Введение

Развитие экологического мониторинга связано с регулярным сбором большого количества измерений, получаемых на разных территориях и в разные моменты времени. Для анализа состояния окружающей среды необходимо хранить не только сами значения показателей, но и сведения о месте измерения, типе показателя, единице измерения, времени получения данных и источнике загрузки. Если такие данные остаются в виде отдельных CSV-файлов, их трудно сопоставлять между собой, фильтровать по периоду, использовать для построения графиков и проверять на целостность.

Актуальность темы курсовой работы определяется необходимостью перехода от разрозненного файлового хранения экологических измерений к сетевой базе данных, доступной через backend API и пользовательское веб-приложение. В рамках проекта рассматривается ситуация, когда данные собираются на экологических полигонах и затем загружаются пользователем в систему в виде CSV-файлов широкого формата. Каждый файл содержит столбец даты и набор столбцов с показателями, например CO2, влажностью, температурой, давлением, освещенностью и уровнем шума. Для дальнейшего анализа такой формат преобразуется в нормализованную таблицу измерений.

Проблема хранения данных только в CSV-файлах заключается в том, что каждый файл фактически является самостоятельным источником. В нем отсутствуют внешние ключи, единая система справочников, журнал загрузок, контроль связей между полигоном, датчиком и единицей измерения. При увеличении числа файлов становится сложно определить, какие данные относятся к одному полигону, кто их загрузил, какие измерения были пропущены, какие значения можно использовать в графиках. Кроме того, прямой анализ файлов не обеспечивает удобной фильтрации по нескольким параметрам и не позволяет эффективно строить временные ряды.

Цель курсовой работы состоит в разработке базы данных MySQL и веб-приложения для хранения, просмотра и анализа экологических измерений, загружаемых из CSV-файлов. В проекте база данных выступает центральным элементом системы: она хранит справочники, журнал импортов и нормализованные измерения, а backend предоставляет прикладной доступ к этим данным через REST API.

Для достижения поставленной цели были определены следующие задачи: проанализировать предметную область экологических измерений; определить требования к структуре базы данных и веб-приложению; выделить сущности предметной области; спроектировать нормализованную схему базы данных; реализовать физическую модель в MySQL с помощью Alembic-миграций; описать взаимодействие FastAPI backend с MySQL через SQLAlchemy ORM; реализовать загрузку CSV-файлов, просмотр измерений, фильтрацию и построение графиков; подготовить примеры SQL-запросов, демонстрирующих работу с сетевой базой данных.

Объектом работы является информационная система экологического мониторинга. Предметом работы является проектирование и реализация сетевой базы данных для хранения экологических измерений, а также механизм взаимодействия этой базы данных с backend API и пользовательским веб-интерфейсом.

В проекте используются следующие технологии: MySQL 8 для хранения данных, FastAPI для реализации backend API, SQLAlchemy ORM для выполнения запросов к базе данных, Alembic для управления миграциями, Angular и TypeScript для пользовательского интерфейса, Apache ECharts для визуализации временных рядов, Docker Compose для запуска сервисов приложения.

Структура отчета соответствует этапам разработки. В первом разделе приведен анализ предметной области и постановка задачи. Во втором разделе описано логическое проектирование базы данных. В третьем разделе рассмотрена физическая реализация схемы в MySQL. В четвертом разделе описана архитектура веб-приложения и работа backend с базой данных. В пятом разделе приведено описание пользовательских страниц приложения с местами для последующей вставки скриншотов. В шестом разделе представлены SQL-запросы, которые показывают основные операции выборки, фильтрации и агрегации экологических измерений.

# 1. Анализ предметной области и постановка задачи

## 1.1. Описание предметной области экологических измерений

Предметная область проекта связана с хранением и анализом измерений, получаемых на экологических полигонах. Под экологическим полигоном в рамках работы понимается территория или точка наблюдения, для которой регулярно фиксируются показатели окружающей среды. В проекте полигон является самостоятельным справочным объектом: у него есть название, местоположение и дополнительное описание. Один полигон может иметь множество измерений и множество загруженных CSV-файлов.

На полигонах собираются значения разных экологических показателей. В реализованном проекте используются типы показателей, соответствующие столбцам CSV-файла и справочнику `sensor_types`: CO2, влажность, температура, давление, освещенность и уровень шума. Эти показатели рассматриваются именно как типы измеряемых величин, а не как отдельные физические устройства. Поэтому в базе данных нет сущностей физических датчиков, таблиц `sensors`, `polygon_sensors` или сущности эксперимента. Такая модель соответствует фактическому назначению приложения: хранить временные ряды значений по полигону, типу показателя и времени измерения.

Сбор данных выполняется человеком, который загружает подготовленный CSV-файл через веб-интерфейс. При загрузке пользователь выбирает полигон и указывает фамилию загрузившего. В backend эта фамилия связывается со справочником `data_collectors`: если запись с такой фамилией уже существует, она используется повторно, если нет, создается новая запись. Такой подход позволяет вести журнал загрузок и связывать импортированные измерения с ответственным лицом без введения сложной подсистемы пользователей.

Для анализа экологических измерений важны временная привязка и возможность сопоставления показателей. Например, пользователь должен иметь возможность посмотреть изменение CO2 и влажности на одном полигоне за выбранный период или сравнить влажность на нескольких полигонах. В терминах базы данных это означает, что измерения должны храниться в такой форме, где каждый факт измерения имеет отдельную строку, а справочные признаки вынесены во внешние таблицы.

Предметная область не требует хранения исходного CSV-файла как основного объекта анализа. CSV-файл является только источником данных и основанием для записи в журнал загрузок. После обработки каждая непустая корректная ячейка значения превращается в отдельную запись таблицы `measurements`. Поэтому аналитические функции приложения строятся не по таблице импортов, а по таблице измерений.

## 1.2. Формат исходных данных

Исходные данные поступают в виде CSV-файлов широкого формата. В таком формате одна строка файла соответствует одному моменту времени, а отдельные столбцы после даты соответствуют разным экологическим показателям. Обязательным столбцом является `Дата`. Остальные столбцы интерпретируются как типы показателей. Пример структуры файла соответствует используемым демонстрационным данным проекта:

```csv
Дата,CO2,Влажность,Температура,Давление,Освещённость,Уровень шума
2025-01-01 00:00:00,521,52,12.3,1005,13,39
2025-01-01 01:00:00,512,,16.9,1008,0,42
```

В приведенном примере первая строка содержит заголовки. Первый столбец задает дату и время измерения, а остальные столбцы содержат значения показателей. Если в ячейке отсутствует значение или значение невозможно преобразовать к числу, оно не записывается в таблицу измерений и учитывается в количестве пропущенных значений. Это поведение важно для практической работы, поскольку реальные CSV-файлы могут содержать неполные строки.

При импорте широкая структура CSV преобразуется в длинный формат. Например, одна строка CSV с шестью заполненными показателями создает шесть строк в таблице `measurements`: по одной строке для CO2, влажности, температуры, давления, освещенности и уровня шума. В каждой такой строке сохраняются идентификатор полигона, идентификатор типа показателя, идентификатор единицы измерения, время измерения, числовое значение и ссылка на запись в журнале импорта.

Поддержка CSV-файлов реализована в backend-сервисе импорта. Проект учитывает разные варианты кодировки файла: UTF-8 с BOM, UTF-8 и CP1251. Для столбца даты поддерживаются несколько форматов даты и времени, включая ISO-подобный формат `YYYY-MM-DD HH:MM:SS` и российский формат `DD.MM.YYYY HH:MM`. Это делает импорт устойчивым к типичным форматам, которые могут встретиться при ручной подготовке данных.

## 1.3. Требования к базе данных

База данных должна обеспечивать надежное хранение справочной и фактической информации. К справочной информации относятся полигоны, сборщики данных, единицы измерения и типы измеряемых показателей. К фактической информации относятся записи журнала импортов и сами измерения. Разделение этих данных позволяет избежать дублирования названий полигонов, названий показателей и единиц измерения в каждой строке измерений.

К базе данных предъявляются следующие требования:

- хранить перечень экологических полигонов;
- хранить сведения о людях, загружающих CSV-файлы;
- хранить единицы измерения и их условные обозначения;
- хранить типы экологических показателей и связывать их с единицами измерения;
- хранить историю загрузок CSV-файлов с указанием полигона, загрузившего, имени файла, статуса и количества обработанных записей;
- хранить измерения в нормализованном длинном формате;
- поддерживать фильтрацию измерений по полигону, типу показателя, периоду и файлу импорта;
- обеспечивать построение графиков по временным рядам;
- обеспечивать сравнение одного показателя на нескольких полигонах;
- быть нормализованной не ниже третьей нормальной формы;
- использовать первичные ключи, внешние ключи и индексы для сохранения целостности и ускорения выборок.

Важным требованием является отсутствие избыточных сущностей. В проекте не вводятся эксперименты и физические датчики, поскольку фактическая структура приложения не хранит сведения о конкретных устройствах. Система работает с типами показателей, а не с инвентарным учетом оборудования. Это упрощает модель и делает ее достаточной для задач курсовой работы.

## 1.4. Требования к веб-приложению

Веб-приложение должно предоставлять пользователю доступ к операциям, которые необходимы для работы с экологическими измерениями. Пользовательский интерфейс реализован на Angular и взаимодействует с backend через HTTP-запросы к FastAPI. Backend, в свою очередь, выполняет операции с MySQL через SQLAlchemy ORM.

К веб-приложению предъявляются следующие требования:

- загрузка CSV-файла с выбором полигона и указанием фамилии загрузившего;
- отображение результата загрузки, включая статус, количество строк CSV, количество созданных измерений и число пропущенных значений;
- просмотр таблицы экологических измерений;
- фильтрация измерений по полигону, типу показателя, файлу импорта и периоду;
- сортировка измерений по времени и постраничный вывод;
- построение графика нескольких показателей на одном полигоне;
- сравнение одного показателя на нескольких полигонах;
- выбор режима агрегации данных для графиков: исходные значения, почасовая агрегация или дневная агрегация;
- просмотр справочников полигонов, типов показателей и единиц измерения;
- просмотр сведений о последних загрузках CSV-файлов.

Функциональность приложения должна быть связана именно с базой данных. Все основные страницы получают данные через backend API, а не из локальных файлов frontend. Это соответствует теме дисциплины «Сетевые базы данных», поскольку база данных размещается как отдельный сервис MySQL, а доступ к ней выполняется через серверное приложение.

# 2. Проектирование базы данных

## 2.1. Выделение основных сущностей предметной области

Логическая модель базы данных строится вокруг факта экологического измерения. Один факт измерения определяется полигоном, типом показателя, единицей измерения, временем и числовым значением. Дополнительно факт может быть связан с CSV-файлом, из которого он был получен. На основании анализа предметной области выделены следующие сущности: `polygons`, `data_collectors`, `measurement_units`, `sensor_types`, `import_files`, `measurements`.

Сущность `polygons` хранит сведения о территориях наблюдения. Сущность `data_collectors` хранит людей, которые загружают CSV-файлы. Сущность `measurement_units` является справочником единиц измерения. Сущность `sensor_types` хранит типы измеряемых показателей и связывает их с единицами измерения. Сущность `import_files` фиксирует факт загрузки CSV-файла. Сущность `measurements` хранит нормализованные значения экологических измерений.

Такое разделение отражает различие между справочными, журнальными и фактическими данными. Справочники изменяются редко, журнал импортов пополняется при каждой загрузке файла, а таблица измерений является наиболее объемной и используется во всех аналитических сценариях.

## 2.2. Описание таблиц базы данных

Таблица `polygons` предназначена для хранения экологических полигонов. В приложении полигон выбирается пользователем перед загрузкой CSV-файла и используется как основной фильтр при просмотре и построении графиков.

| location | VARCHAR(255) | Местоположение полигона, может отсутствовать. |
| description | TEXT | Текстовое описание полигона. |
| created_at | DATETIME | Дата и время создания записи. |

Таблица `data_collectors` предназначена для хранения сведений о людях, которые загружают CSV-файлы. В текущей версии приложения обязательным является указание фамилии, имя и отчество являются необязательными.

Таблица 2 — Структура таблицы data_collectors

Таблица `measurement_units` является справочником единиц измерения. Она позволяет хранить единицы централизованно и связывать их с типами показателей и с конкретными измерениями.

Таблица 3 — Структура таблицы measurement_units

| Имя поля | Тип данных | Назначение |
|---|---|---|

## 2.3. Главная таблица measurements

Главной таблицей проекта является `measurements`, поскольку именно она содержит факты экологических наблюдений. Все основные аналитические сценарии приложения обращаются к этой таблице: просмотр измерений, фильтрация по полигону и показателю, построение графиков, сравнение полигонов и агрегация по времени. Справочники используются для расшифровки идентификаторов, а журнал импортов используется для аудита происхождения данных, но сами значения находятся в `measurements`.

Структура таблицы `measurements` соответствует модели временных рядов. Каждая строка описывает одно числовое значение. В строке хранится время измерения, значение и набор внешних ключей. Такой формат удобен для SQL-запросов, потому что фильтр по полигону, типу показателя и периоду выражается обычными условиями `WHERE`, а построение графиков выполняется выборкой отсортированного набора точек.

Использование длинного формата также упрощает расширение набора показателей. Если в будущем потребуется добавить новый тип экологического показателя, достаточно внести запись в справочник `sensor_types` и связать ее с единицей измерения. Структура таблицы `measurements` при этом не изменяется. В широком формате потребовалось бы добавлять новый столбец, что приводило бы к усложнению миграций и запросов.

Поле `unit_id` в таблице `measurements` дублирует связь, которая уже существует между `sensor_types` и `measurement_units`. В текущем проекте это сделано осознанно: запись измерения сохраняет единицу измерения на момент импорта и упрощает выборки для таблиц и графиков. При этом целостность поддерживается внешним ключом на справочник единиц измерения.

## 2.4. Таблица import_files как журнал загрузок

Таблица `import_files` является журналом загрузок, а не основной таблицей анализа. Ее назначение состоит в фиксации информации о том, какой CSV-файл был загружен, кем, для какого полигона, когда и с каким результатом. Она хранит технические и учетные сведения об импорте, но не заменяет таблицу измерений.

Разделение таблиц `import_files` и `measurements` позволяет хранить историю происхождения данных без смешивания ее с фактическими экологическими значениями. Одна запись в `import_files` может быть связана с большим количеством строк в `measurements`. Например, если CSV-файл содержит 100 строк и 6 заполненных столбцов показателей, после импорта может быть создано около 600 измерений, но запись о файле останется одной.

Такой подход удобен для аудита. Пользователь может определить, из какого файла было получено конкретное измерение, сколько строк было обработано при загрузке, сколько значений было пропущено и завершилась ли загрузка успешно. Если импорт завершился ошибкой, поле `status` получает значение `failed`, а поле `error_message` сохраняет текст ошибки. При успешной обработке статус меняется на `processed`.

## 2.5. Связи между таблицами

Связи между таблицами отражают логику предметной области. Один полигон может иметь множество измерений, поэтому между `polygons` и `measurements` установлена связь один-ко-многим. Один тип показателя может встречаться во множестве измерений, поэтому между `sensor_types` и `measurements` также установлена связь один-ко-многим. Одна единица измерения может использоваться несколькими типами показателей и множеством измерений.

Журнал импортов связан с полигонами и сборщиками данных. Один полигон может иметь множество загруженных CSV-файлов, а один сборщик данных может загрузить множество файлов. Поэтому таблица `import_files` содержит внешние ключи `polygon_id` и `collector_id`. Измерения, созданные при импорте, ссылаются на запись `import_files` через поле `import_file_id`.

Основные связи можно представить следующим образом:

- `polygons 1:N measurements`;
- `sensor_types 1:N measurements`;
- `measurement_units 1:N sensor_types`;
- `measurement_units 1:N measurements`;
- `import_files 1:N measurements`;
- `polygons 1:N import_files`;
- `data_collectors 1:N import_files`.

Все связи реализованы внешними ключами на уровне схемы базы данных. Это позволяет MySQL контролировать корректность ссылок и не допускать появления измерения, указывающего на несуществующий полигон, тип показателя или единицу измерения.

Это должно быть отражено в диаграмме.

## 2.7. Нормализация базы данных до третьей нормальной формы

Спроектированная база данных соответствует требованиям третьей нормальной формы. Первая нормальная форма обеспечивается тем, что в таблицах отсутствуют повторяющиеся группы столбцов, а каждое поле содержит атомарное значение. Широкий CSV-файл не переносится в базу данных как набор столбцов `CO2`, `Влажность`, `Температура` и так далее. Вместо этого каждая ячейка показателя становится отдельной строкой таблицы `measurements`.

Вторая нормальная форма обеспечивается использованием простых первичных ключей и зависимостью неключевых атрибутов от ключа своей таблицы. Например, в таблице `polygons` название и местоположение зависят от `polygon_id`, в таблице `sensor_types` код и название зависят от `sensor_type_id`, а в таблице `measurements` время и значение относятся к конкретному `measurement_id`.

Третья нормальная форма обеспечивается отсутствием транзитивных зависимостей неключевых атрибутов внутри одной таблицы. Название полигона не хранится в `measurements`, а вынесено в `polygons`. Название типа показателя не хранится в `measurements`, а хранится в `sensor_types`. Символ единицы измерения хранится в `measurement_units`. Благодаря этому при изменении названия полигона или описания единицы измерения не требуется обновлять множество строк измерений.

Отдельного внимания требует поле `unit_id` в таблице `measurements`. С точки зрения строгой логической модели единица измерения может быть получена через `sensor_type_id`. Однако в реализованной базе это поле используется как сохранение единицы измерения, примененной к фактическому измерению, и как практическое упрощение выборок. Поскольку поле также связано внешним ключом со справочником единиц измерения, оно не превращается в текстовое дублирование справочника и не нарушает целостность данных.

## 2.8. Индексы для ускорения фильтрации и построения графиков

Таблица `measurements` потенциально является самой большой таблицей базы данных. Поэтому для нее в миграции созданы составные индексы, соответствующие основным сценариям выборки. Индекс `idx_measurements_polygon_sensor_time` строится по полям `polygon_id`, `sensor_type_id`, `measured_at` и ускоряет фильтрацию измерений по одному полигону, одному или нескольким типам показателей и периоду. Такой сценарий используется при просмотре таблицы измерений и построении графика нескольких показателей на одном полигоне.

Индекс `idx_measurements_sensor_time_polygon` строится по полям `sensor_type_id`, `measured_at`, `polygon_id`. Он полезен для сравнения одного показателя на нескольких полигонах. В этом режиме сначала выбирается тип показателя, затем временной интервал, а полигон используется для группировки или разделения серий графика.

Дополнительно создан индекс `idx_measurements_time` по полю `measured_at`. Он ускоряет выборки, где основным условием является временной интервал. Индекс `idx_measurements_import_file` ускоряет получение измерений, связанных с конкретным CSV-файлом. Для журнала импортов создан индекс `idx_import_files_polygon_uploaded`, который ускоряет просмотр загрузок по полигону и дате.

Использование индексов особенно важно для веб-приложения, поскольку пользователь ожидает быстрый отклик при применении фильтров и построении графиков. Без индексов база данных должна была бы выполнять полный просмотр таблицы измерений, что при увеличении объема данных могло бы привести к заметным задержкам.

# 3. Физическая реализация базы данных в MySQL

## 3.1. Общая структура физической модели

Физическая реализация базы данных выполнена для MySQL 8. В проекте база данных запускается как отдельный сервис Docker Compose с образом `mysql:8.0`. Backend подключается к этому сервису по сетевому имени контейнера и использует параметры подключения из переменных окружения. Такая схема соответствует сетевой архитектуре: база данных отделена от серверного приложения и доступна через сетевое соединение.

Таблицы создаются через Alembic-миграцию `20260524_01_init_schema.py`. Миграция описывает создание всех шести таблиц, первичных ключей, внешних ключей, уникальных ограничений и индексов. Использование Alembic позволяет воспроизводимо развертывать структуру базы данных в разных окружениях и отслеживать изменения схемы.

В MySQL 8 для транзакционных таблиц по умолчанию используется механизм хранения InnoDB. Для данной базы это важно, поскольку InnoDB поддерживает внешние ключи, транзакции и блокировки на уровне строк. Эти свойства нужны для согласованной загрузки CSV-файлов: запись в журнал импорта и добавление измерений должны выполняться как связанная операция.

Физическая модель соответствует логической схеме: справочники вынесены в отдельные таблицы, журнал импортов связан с полигонами и сборщиками, а таблица измерений содержит ссылки на все необходимые справочные сущности. При этом в базе используются типы данных, подходящие для MySQL и SQLAlchemy.

## 3.2. Первичные и внешние ключи

Во всех таблицах используются числовые первичные ключи типа `BIGINT` с автоинкрементом. Такой тип выбран потому, что таблица измерений может быстро расти при импорте CSV-файлов: одна строка файла создает несколько строк измерений. Использование `BIGINT` дает значительный запас по количеству записей.

Первичные ключи имеют следующие поля: `polygon_id`, `collector_id`, `unit_id`, `sensor_type_id`, `import_file_id`, `measurement_id`. Все они создаются как `PRIMARY KEY` и автоматически увеличиваются при добавлении новых записей. В SQLAlchemy для совместимости с тестовым SQLite используется вариант типа, который в MySQL работает как `BigInteger`, а в SQLite как `Integer`.

Внешние ключи реализуют связи между таблицами. Таблица `sensor_types` содержит `unit_id`, ссылающийся на `measurement_units`. Таблица `import_files` содержит `polygon_id` и `collector_id`, ссылающиеся на `polygons` и `data_collectors`. Таблица `measurements` содержит `polygon_id`, `sensor_type_id`, `unit_id` и `import_file_id`. Поле `import_file_id` допускает значение `NULL`, чтобы структура могла поддерживать измерения, добавленные не через CSV-импорт, хотя в текущем пользовательском сценарии данные создаются именно при загрузке CSV.

## 3.3. Типы данных, используемые в таблицах

Для идентификаторов используется `BIGINT`. Для названий, кодов и статусов используются строковые типы `VARCHAR`. Например, название полигона хранится как `VARCHAR(255)`, код типа показателя как `VARCHAR(100)`, символ единицы измерения как `VARCHAR(50)`, а статус импорта как `VARCHAR(50)`.

Для текстовых описаний и сообщений об ошибках используется тип `TEXT`. Он применяется там, где длина значения заранее не ограничивается короткой строкой: описание полигона, описание единицы измерения, описание типа показателя и сообщение об ошибке импорта.

Для дат и времени используется тип `DATETIME`. Он применяется в полях `created_at`, `updated_at`, `uploaded_at` и `measured_at`. Важнейшим полем для аналитики является `measured_at`, потому что по нему выполняются фильтрация, сортировка, построение временных рядов и агрегация.

Для числового значения измерения используется `DECIMAL(16,4)`. Этот тип выбран вместо вещественного `FLOAT`, поскольку экологические показатели часто требуют предсказуемого хранения десятичных значений. Масштаб 4 знака после запятой достаточен для показателей проекта, а точность 16 дает запас для больших значений.

## 3.4. Реализация связей между таблицами

Связи реализованы как в Alembic-миграции, так и в SQLAlchemy-моделях. В миграции используются `sa.ForeignKeyConstraint`, а в моделях - `ForeignKey` и `relationship`. Благодаря этому база данных контролирует целостность ссылок, а ORM позволяет удобно обращаться к связанным объектам.

Например, у модели `Measurement` есть связи `polygon`, `sensor_type`, `unit` и `import_file`. При получении измерений backend может присоединять справочники через SQLAlchemy `join` и формировать ответ API, в котором присутствуют не только идентификаторы, но и человекочитаемые названия полигона, датчика и единицы измерения.

Для журнала импортов реализованы связи с `Polygon` и `DataCollector`. Поэтому endpoint `/api/imports` может вернуть имя файла, название полигона, фамилию загрузившего, время загрузки, статус и количество созданных измерений. Эти данные не дублируются в одной таблице, а собираются из связанных таблиц.

## 3.5. Ограничения целостности данных

В базе данных используются ограничения `NOT NULL` для обязательных полей. Например, у измерения обязательны `polygon_id`, `sensor_type_id`, `unit_id`, `measured_at` и `value`. У файла импорта обязательны `polygon_id`, `collector_id`, `file_name`, `uploaded_at`, `rows_count`, `measurements_count` и `status`. Это предотвращает появление неполных записей, которые нельзя корректно использовать в аналитике.

Уникальные ограничения применяются к полям `measurement_units.symbol` и `sensor_types.code`. Уникальность символа единицы измерения предотвращает появление нескольких записей с одинаковым обозначением, например `%` или `ppm`. Уникальность кода типа показателя нужна для обработки CSV-колонок: backend сопоставляет название столбца с кодом показателя и ожидает, что такой код однозначно определяет запись справочника.

Дополнительная целостность обеспечивается прикладной логикой backend. Перед импортом CSV проверяется наличие полигона. Для фамилии загрузившего выполняется поиск существующей записи без учета регистра. CSV-файл проверяется на наличие имени, расширения `.csv`, непустого содержимого, столбца даты и известных столбцов показателей. Некорректные числовые значения не записываются в `measurements`, а учитываются как пропущенные.

# 4. Архитектура веб-приложения

## 4.1. Общая схема работы приложения

Архитектура проекта построена по схеме Angular frontend -> FastAPI backend -> SQLAlchemy ORM -> MySQL. Пользователь работает с веб-интерфейсом Angular. Frontend отправляет HTTP-запросы на backend API по адресу `http://localhost:8000/api`. Backend обрабатывает запросы, выполняет операции с базой данных через SQLAlchemy и возвращает JSON-ответы. MySQL хранит постоянные данные приложения.

Такая архитектура разделяет ответственность между слоями. Frontend отвечает за отображение страниц, форм, таблиц и графиков. Backend отвечает за проверку входных данных, бизнес-логику импорта CSV, выполнение запросов и подготовку ответов. База данных отвечает за хранение, связи, индексы и целостность данных.

В проекте предусмотрены следующие группы API: справочники, импорт CSV, измерения, графики, Dashboard и health-check. Роутеры подключаются к общему `api_router`, а затем включаются в FastAPI-приложение с префиксом `/api`. Это обеспечивает единое пространство endpoint-ов.

## 4.2. Использованные технологии и фреймворки

Backend реализован на Python с использованием FastAPI. Этот фреймворк позволяет описывать HTTP-endpoint-ы, параметры запросов, формы загрузки файлов и модели ответов. Для работы с базой данных используется SQLAlchemy 2.0, а для миграций - Alembic. Драйвер подключения к MySQL - PyMySQL.

Frontend реализован на Angular 18 и TypeScript. В проекте используются standalone-компоненты, Angular Router, HttpClient и Reactive Forms. Для построения графиков используется Apache ECharts. Библиотека подключается динамически на странице графиков, после чего создается экземпляр графика и обновляется набор серий.

Запуск инфраструктуры описан в `docker-compose.yml`. В нем определены сервисы `mysql` и `backend`. MySQL запускается с образом `mysql:8.0`, а backend собирается из каталога `backend` и запускает `uvicorn app.main:app`. Frontend запускается отдельно командой `npm start` из каталога `frontend`.

## 4.3. Структура проекта

Репозиторий имеет монорепозиторную структуру. Backend расположен в каталоге `backend/app`. В нем находятся модели базы данных, схемы Pydantic, роутеры, сервисы, конфигурация и подключение к базе данных. Миграции Alembic расположены в каталоге `backend/alembic/versions`.

Frontend расположен в каталоге `frontend/src/app`. В нем есть компоненты страниц Dashboard, загрузки CSV, просмотра измерений, графиков и справочников. Общий сервис `ApiService` инкапсулирует HTTP-запросы к backend. Типы ответов API описаны в `api.models.ts`.

Демонстрационные CSV-файлы находятся в каталоге `sample_data`. В них используются столбцы `Дата`, `CO2`, `Влажность`, `Температура`, `Давление`, `Освещённость`, `Уровень шума`. Эти данные соответствуют поддерживаемым типам показателей в backend-сервисе импорта.

## 4.4. Подключение backend к базе данных MySQL

Подключение backend к MySQL реализовано в файлах `backend/app/core/config.py` и `backend/app/database.py`. В конфигурации собирается строка подключения SQLAlchemy с драйвером `mysql+pymysql`, именем хоста, портом, названием базы данных, пользователем и паролем. В файле `database.py` создаются `engine`, фабрика сессий `SessionLocal` и зависимость `get_db`, которая используется в FastAPI-роутерах.

Листинг 1 — Подключение backend к базе данных MySQL

```python
@property
def sqlalchemy_database_uri(self) -> str:
    return (
        f"mysql+pymysql://{self.db_user}:{self.db_password}"
        f"@{self.db_host}:{self.db_port}/{self.db_name}?charset=utf8mb4"
    )

engine = create_engine(
    settings.sqlalchemy_database_uri,
    pool_pre_ping=True,
    future=True,
)

SessionLocal = sessionmaker(
    autocommit=False,
    autoflush=False,
    bind=engine,
    class_=Session,
)

def get_db() -> Generator[Session, None, None]:
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

В листинге показано, что backend не обращается к базе данных напрямую через низкоуровневый драйвер. Вместо этого создается SQLAlchemy engine и фабрика сессий. Параметр `pool_pre_ping=True` помогает проверять соединение перед использованием, что полезно при работе с сетевой базой данных в контейнерной среде. Зависимость `get_db` открывает сессию на время обработки запроса и закрывает ее после завершения.

## 4.5. Выполнение запросов к базе данных через ORM

В проекте запросы к базе данных выполняются через SQLAlchemy ORM и выражения `select`. Такой подход позволяет описывать запросы на Python, сохраняя связь с моделями таблиц. В роутерах используются зависимости FastAPI: объект `Session` передается в endpoint через `Depends(get_db)`.

ORM-запросы в проекте выполняют несколько типов операций. Для справочников используются простые выборки с сортировкой, например `select(Polygon).order_by(Polygon.name.asc())`. Для таблицы измерений используется запрос с несколькими `join`, поскольку ответ должен содержать название полигона, название и код показателя, символ единицы измерения, имя файла и фамилию загрузившего. Для графиков используются запросы с фильтрацией, сортировкой и, при необходимости, агрегацией через `avg`.

Использование ORM не отменяет SQL-логику. Напротив, SQLAlchemy генерирует SQL-запросы, соответствующие структуре базы данных. Поэтому в отчете отдельно приведены SQL-эквиваленты основных сценариев выборки.

## 4.6. Пример получения данных из базы для таблицы измерений

Endpoint `/api/measurements` возвращает постраничный список измерений. Он поддерживает фильтры по полигону, типу показателя, периоду, загрузившему и файлу импорта. В ответе возвращаются не только идентификаторы, но и названия из справочников, что удобно для отображения в Angular-таблице.

Листинг 2 — ORM-запрос для получения таблицы измерений

```python
base_stmt = (
    select(
        Measurement.measurement_id.label("measurement_id"),
        Measurement.measured_at.label("measured_at"),
        Polygon.name.label("polygon_name"),
        SensorType.name.label("sensor_name"),
        SensorType.code.label("sensor_code"),
        Measurement.value.label("value"),
        MeasurementUnit.symbol.label("unit_symbol"),
        import_alias.file_name.label("file_name"),
        collector_alias.last_name.label("collector_last_name"),
    )
    .join(Polygon, Polygon.polygon_id == Measurement.polygon_id)
    .join(SensorType, SensorType.sensor_type_id == Measurement.sensor_type_id)
    .join(MeasurementUnit, MeasurementUnit.unit_id == Measurement.unit_id)
    .outerjoin(import_alias, import_alias.import_file_id == Measurement.import_file_id)
    .outerjoin(collector_alias, collector_alias.collector_id == import_alias.collector_id)
)

if filters:
    base_stmt = base_stmt.where(and_(*filters))

rows = db.execute(
    base_stmt.order_by(order_expression).limit(limit).offset(offset)
).mappings().all()
```

В этом запросе таблица `measurements` является основной. К ней присоединяются `polygons`, `sensor_types`, `measurement_units`, `import_files` и `data_collectors`. Внешние соединения используются для файла импорта и загрузившего, потому что поле `import_file_id` в измерениях допускает `NULL`. Результат используется для формирования массива элементов таблицы на frontend.

## 4.7. Пример получения данных из базы для построения графика

Для построения графика нескольких показателей на одном полигоне используется endpoint `/api/charts/multi-sensor`. Он принимает идентификатор полигона, список идентификаторов типов показателей, период и режим агрегации. Если выбран режим `raw`, возвращаются исходные значения. Если выбран режим `hourly` или `daily`, запрос группирует данные по временному интервалу и вычисляет среднее значение.

Листинг 3 — ORM-запрос для построения графика

```python
bucket = _bucket_expression(db, aggregation).label("bucket")
value_expr = Measurement.value if aggregation == "raw" else func.avg(Measurement.value)

stmt = (
    select(
        SensorType.sensor_type_id.label("sensor_type_id"),
        SensorType.name.label("sensor_name"),
        bucket,
        value_expr.label("value"),
    )
    .join(SensorType, SensorType.sensor_type_id == Measurement.sensor_type_id)
    .where(and_(
        Measurement.polygon_id == polygon_id,
        Measurement.sensor_type_id.in_(sensor_ids),
    ))
)

if aggregation != "raw":
    stmt = stmt.group_by(SensorType.sensor_type_id, SensorType.name, bucket)

stmt = stmt.order_by(bucket.asc(), SensorType.sensor_type_id.asc())
rows = db.execute(stmt).mappings().all()
```

Код показывает, что график строится непосредственно по таблице `measurements`. Для MySQL при почасовой и дневной агрегации применяется функция `date_format`, а для тестового SQLite предусмотрен вариант через `strftime`. После выполнения запроса backend группирует строки по типам показателей и возвращает серии данных, которые затем отображаются в Apache ECharts.

# 6. SQL-запросы к базе данных

## 6.1. Запрос для вывода измерений с данными из справочников

Назначение запроса - вывести измерения вместе с данными из справочников, которые нужны для отображения таблицы в интерфейсе. Используются таблицы `measurements`, `polygons`, `sensor_types`, `measurement_units`, `import_files` и `data_collectors`.

```sql
SELECT
    m.measurement_id,
    m.measured_at,
    p.name AS polygon_name,
    st.name AS sensor_name,
    st.code AS sensor_code,
    m.value,
    mu.symbol AS unit_symbol,
    i.file_name,
    dc.last_name AS collector_last_name
FROM measurements AS m
JOIN polygons AS p
    ON p.polygon_id = m.polygon_id
JOIN sensor_types AS st
    ON st.sensor_type_id = m.sensor_type_id
JOIN measurement_units AS mu
    ON mu.unit_id = m.unit_id
LEFT JOIN import_files AS i
    ON i.import_file_id = m.import_file_id
LEFT JOIN data_collectors AS dc
    ON dc.collector_id = i.collector_id
ORDER BY m.measured_at DESC
LIMIT 100 OFFSET 0;
```

Результатом является набор строк измерений с человекочитаемыми названиями полигона и показателя, символом единицы измерения, именем файла и фамилией загрузившего. Такой запрос соответствует таблице измерений в пользовательском интерфейсе.

## 6.2. Запрос для фильтрации измерений по полигону и датчику

Назначение запроса - получить значения одного показателя на одном полигоне за выбранный период. Используются таблицы `measurements`, `polygons`, `sensor_types` и `measurement_units`.

```sql
SELECT
    m.measured_at,
    p.name AS polygon_name,
    st.name AS sensor_name,
    m.value,
    mu.symbol AS unit_symbol
FROM measurements AS m
JOIN polygons AS p
    ON p.polygon_id = m.polygon_id
JOIN sensor_types AS st
    ON st.sensor_type_id = m.sensor_type_id
JOIN measurement_units AS mu
    ON mu.unit_id = m.unit_id
WHERE m.polygon_id = :polygon_id
  AND m.sensor_type_id = :sensor_type_id
  AND m.measured_at BETWEEN :date_from AND :date_to
ORDER BY m.measured_at ASC;
```

Запрос возвращает временной ряд значений. Он используется для просмотра отфильтрованных данных и может быть основой для построения графика одного показателя.

## 6.3. Запрос для построения графика нескольких датчиков на одном полигоне

Назначение запроса - получить данные для нескольких показателей на одном полигоне. Используются таблицы `measurements`, `sensor_types` и `measurement_units`.

```sql
SELECT
    m.measured_at,
    st.sensor_type_id,
    st.name AS sensor_name,
    st.code AS sensor_code,
    m.value,
    mu.symbol AS unit_symbol
FROM measurements AS m
JOIN sensor_types AS st
    ON st.sensor_type_id = m.sensor_type_id
JOIN measurement_units AS mu
    ON mu.unit_id = m.unit_id
WHERE m.polygon_id = :polygon_id
  AND m.sensor_type_id IN (:sensor_type_id_1, :sensor_type_id_2, :sensor_type_id_3)
  AND m.measured_at BETWEEN :date_from AND :date_to
ORDER BY m.measured_at ASC, st.sensor_type_id ASC;
```

Результат содержит точки временных рядов для нескольких показателей. На backend или frontend строки группируются по `sensor_type_id`, после чего каждая группа отображается как отдельная серия графика.

## 6.4. Запрос для сравнения одного датчика на нескольких полигонах

Назначение запроса - сравнить один показатель на нескольких полигонах за один и тот же период. Используются таблицы `measurements`, `polygons`, `sensor_types` и `measurement_units`.

```sql
SELECT
    m.measured_at,
    p.polygon_id,
    p.name AS polygon_name,
    st.name AS sensor_name,
    m.value,
    mu.symbol AS unit_symbol
FROM measurements AS m
JOIN polygons AS p
    ON p.polygon_id = m.polygon_id
JOIN sensor_types AS st
    ON st.sensor_type_id = m.sensor_type_id
JOIN measurement_units AS mu
    ON mu.unit_id = m.unit_id
WHERE m.sensor_type_id = :sensor_type_id
  AND m.polygon_id IN (:polygon_id_1, :polygon_id_2, :polygon_id_3)

ORDER BY m.measured_at ASC, p.polygon_id ASC;
```

Результат используется для графика сравнения полигонов. Каждая серия строится по одному полигону, а значения относятся к одному выбранному типу показателя.

## 6.5. Запрос для просмотра истории загрузок CSV-файлов

Назначение запроса - вывести журнал импортов CSV-файлов. Используются таблицы `import_files`, `polygons` и `data_collectors`.

```sql
SELECT
    i.import_file_id,
    i.file_name,
    p.name AS polygon_name,
    dc.last_name AS collector_last_name,
    i.uploaded_at,
    i.rows_count,
    i.measurements_count,
    i.status,
    i.error_message
FROM import_files AS i
JOIN polygons AS p
    ON p.polygon_id = i.polygon_id
JOIN data_collectors AS dc
    ON dc.collector_id = i.collector_id
ORDER BY i.uploaded_at DESC;
```

Запрос возвращает историю загрузок: имя файла, полигон, загрузившего, дату, количество строк, количество созданных измерений, статус и сообщение об ошибке. Эти данные используются для блока последних загрузок и для контроля результатов импорта.

## 6.6. Запрос для агрегации измерений по времени

Назначение запроса - агрегировать измерения по часам или дням для построения более сглаженных графиков. Используются таблицы `measurements`, `sensor_types` и `measurement_units`. Ниже приведен пример почасовой агрегации для MySQL.

```sql
SELECT
    DATE_FORMAT(m.measured_at, '%Y-%m-%d %H:00:00') AS time_bucket,
    st.sensor_type_id,
    st.name AS sensor_name,
    AVG(m.value) AS avg_value,
    mu.symbol AS unit_symbol
FROM measurements AS m
JOIN sensor_types AS st
    ON st.sensor_type_id = m.sensor_type_id
JOIN measurement_units AS mu
    ON mu.unit_id = m.unit_id
WHERE m.polygon_id = :polygon_id
  AND m.sensor_type_id IN (:sensor_type_id_1, :sensor_type_id_2)
  AND m.measured_at BETWEEN :date_from AND :date_to
GROUP BY
    DATE_FORMAT(m.measured_at, '%Y-%m-%d %H:00:00'),
    st.sensor_type_id,
    st.name,
    mu.symbol
ORDER BY time_bucket ASC, st.sensor_type_id ASC;
```

Запрос возвращает средние значения по часовым интервалам. Для дневной агрегации выражение `DATE_FORMAT` может быть заменено на формат `'%Y-%m-%d 00:00:00'`. В проекте аналогичная логика реализована в backend-функции выбора временного интервала для графиков.

# Заключение

В ходе курсовой работы была достигнута поставленная цель: разработана база данных MySQL и веб-приложение для хранения, просмотра и анализа экологических измерений. Система позволяет загружать данные из CSV-файлов, преобразовывать широкий формат исходных данных в нормализованный длинный формат и использовать сохраненные измерения для таблиц, фильтрации и графиков.

Была спроектирована структура базы данных, включающая таблицы `polygons`, `data_collectors`, `measurement_units`, `sensor_types`, `import_files` и `measurements`. Центральной таблицей является `measurements`, поскольку она хранит факты экологических измерений и используется во всех аналитических сценариях. Таблица `import_files` выполняет роль журнала загрузок и обеспечивает связь измерений с исходными CSV-файлами.

Физическая реализация базы данных выполнена в MySQL 8. Схема создается через Alembic-миграцию, в которой определены первичные ключи, внешние ключи, уникальные ограничения и индексы. Для идентификаторов используется `BIGINT AUTO_INCREMENT`, для числовых значений - `DECIMAL(16,4)`, для дат и времени - `DATETIME`. Индексы подобраны с учетом основных запросов: фильтрации по полигону, типу показателя и времени, сравнения полигонов и просмотра истории импортов.

База данных приведена к третьей нормальной форме. Справочные данные вынесены в отдельные таблицы, в таблице измерений не дублируются названия полигонов, показателей и единиц измерения, а широкий CSV-формат преобразуется в структуру без повторяющихся групп столбцов. Это повышает целостность данных и упрощает расширение системы.

Веб-приложение реализовано по архитектуре Angular frontend -> FastAPI backend -> SQLAlchemy ORM -> MySQL. Реализованы загрузка CSV, просмотр измерений, фильтрация, построение графика нескольких показателей на одном полигоне, сравнение одного показателя на нескольких полигонах, просмотр справочников и отображение сведений о загрузках. Приложение демонстрирует работу с сетевой базой данных через backend API и соответствует задачам дисциплины «Сетевые базы данных».
